# ESLint Extended Configuration Guide

## Overview

The `.eslintrc.extended.json` file provides comprehensive, strict TypeScript and NestJS-specific linting rules for the backend application. This configuration enforces type safety, code quality, and consistent patterns across the codebase.

## Usage

### Using the Extended Configuration

To use this configuration instead of the default one:

```bash
# In package.json, update the lint script:
"lint": "eslint \"{src,apps,libs,test}/**/*.ts\" --config .eslintrc.extended.json --fix"

# Or run directly:
npx eslint src --config .eslintrc.extended.json --fix
```

### Migration Strategy

1. **Start with specific directories:**
   ```bash
   npx eslint src/modules/user --config .eslintrc.extended.json --fix
   ```

2. **Gradually expand:**
   - Fix one module at a time
   - Review automated fixes before committing
   - Address warnings manually

3. **Full codebase enforcement:**
   ```bash
   npm run lint
   ```

## Key Rule Categories

### 1. Strict Type Safety

#### No Any Type (`@typescript-eslint/no-explicit-any`)
```typescript
// ❌ Bad
function process(data: any) {
  return data.value;
}

// ✅ Good
function process<T extends { value: string }>(data: T): string {
  return data.value;
}

// ✅ Good - Use unknown for truly unknown types
function process(data: unknown): string {
  if (typeof data === 'object' && data !== null && 'value' in data) {
    return String(data.value);
  }
  throw new Error('Invalid data');
}
```

#### Explicit Function Return Types (`@typescript-eslint/explicit-function-return-type`)
```typescript
// ❌ Bad
function getUser(id: string) {
  return userRepository.findOne(id);
}

// ✅ Good
function getUser(id: string): Promise<User | null> {
  return userRepository.findOne(id);
}

// ✅ Good - Arrow function with explicit type
const getUser = (id: string): Promise<User | null> => {
  return userRepository.findOne(id);
};
```

### 2. Naming Conventions

#### NestJS Class Suffixes
```typescript
// ❌ Bad
class User {
  // Missing proper suffix
}

// ✅ Good - Services
@Injectable()
class UserService {
  // ...
}

// ✅ Good - Controllers
@Controller('users')
class UserController {
  // ...
}

// ✅ Good - Guards
@Injectable()
class JwtAuthGuard implements CanActivate {
  // ...
}
```

#### Constant Naming
```typescript
// ❌ Bad
const maxRetries = 3;

// ✅ Good
const MAX_RETRIES = 3;

// ✅ Good - Camel case for non-constants
const userRepository = new UserRepository();
```

#### Enum Naming
```typescript
// ❌ Bad
enum status {
  active = 'active',
  inactive = 'inactive'
}

// ✅ Good
enum UserStatus {
  ACTIVE = 'ACTIVE',
  INACTIVE = 'INACTIVE'
}
```

#### No 'I' Prefix for Interfaces
```typescript
// ❌ Bad
interface IUser {
  name: string;
}

// ✅ Good
interface User {
  name: string;
}

// ✅ Good - Type aliases
type UserResponse = {
  data: User;
  meta: Meta;
};
```

### 3. Unused Variables & Imports

#### No Unused Imports (`unused-imports/no-unused-imports`)
```typescript
// ❌ Bad
import { Injectable, Logger } from '@nestjs/common';
import { User } from './user.entity';

@Injectable()
class UserService {
  // Logger and User are not used
}

// ✅ Good
import { Injectable } from '@nestjs/common';

@Injectable()
class UserService {
  // Only imported what's needed
}
```

#### Unused Variables with Underscore Exception
```typescript
// ❌ Bad
function processUsers(users: User[]) {
  const count = users.length; // count is never used
  return users.map(u => u.name);
}

// ✅ Good - Use underscore for intentionally unused
function processUsers(users: User[], _metadata: Metadata) {
  return users.map(u => u.name);
}

// ✅ Good - Remove if not needed
function processUsers(users: User[]) {
  return users.map(u => u.name);
}
```

### 4. Prefer Const & Let

#### Prefer Const (`prefer-const`)
```typescript
// ❌ Bad
let user = await this.userRepository.findOne(id);
return user;

// ✅ Good
const user = await this.userRepository.findOne(id);
return user;

// ✅ Good - Use let when reassigning
let count = 0;
for (const user of users) {
  count++;
}
```

#### No Var (`no-var`)
```typescript
// ❌ Bad
var userId = 123;

// ✅ Good
const userId = 123;
```

### 5. No Console in Production

#### Use Logger Instead (`no-console`)
```typescript
// ❌ Bad
@Injectable()
class UserService {
  createUser(data: CreateUserDto): Promise<User> {
    console.log('Creating user:', data);
    return this.userRepository.save(data);
  }
}

// ✅ Good - Use NestJS Logger
@Injectable()
class UserService {
  private readonly logger = new Logger(UserService.name);

  createUser(data: CreateUserDto): Promise<User> {
    this.logger.log('Creating user', { userId: data.email });
    return this.userRepository.save(data);
  }
}
```

### 6. NestJS-Specific Patterns

#### Explicit Accessibility Modifiers
```typescript
// ❌ Bad
@Injectable()
class UserService {
  userRepository: Repository<User>;

  async getUser(id: string) {
    return this.userRepository.findOne(id);
  }
}

// ✅ Good
@Injectable()
class UserService {
  private readonly userRepository: Repository<User>;

  public async getUser(id: string): Promise<User | null> {
    return this.userRepository.findOne(id);
  }
}
```

#### Constructor Parameter Properties
```typescript
// ❌ Bad
@Injectable()
class UserService {
  private userRepository: Repository<User>;

  constructor(userRepository: Repository<User>) {
    this.userRepository = userRepository;
  }
}

// ✅ Good - Parameter property shorthand
@Injectable()
class UserService {
  constructor(
    private readonly userRepository: Repository<User>,
  ) {}
}
```

#### Member Ordering
```typescript
// ✅ Good - Consistent ordering
@Injectable()
class UserService {
  // 1. Static fields
  private static readonly DEFAULT_LIMIT = 10;

  // 2. Instance fields
  private readonly logger = new Logger(UserService.name);

  // 3. Constructor
  constructor(
    private readonly userRepository: Repository<User>,
  ) {}

  // 4. Public methods
  public async findAll(): Promise<User[]> {
    return this.userRepository.find();
  }

  // 5. Private methods
  private validateUser(user: User): boolean {
    return user.email !== null;
  }
}
```

### 7. Promise & Async Handling

#### No Floating Promises
```typescript
// ❌ Bad
@Injectable()
class UserService {
  async deleteUser(id: string): Promise<void> {
    this.logAudit(id); // Floating promise!
  }

  private async logAudit(id: string): Promise<void> {
    await this.auditLog.create({ action: 'delete', userId: id });
  }
}

// ✅ Good - Await the promise
@Injectable()
class UserService {
  async deleteUser(id: string): Promise<void> {
    await this.logAudit(id);
  }

  private async logAudit(id: string): Promise<void> {
    await this.auditLog.create({ action: 'delete', userId: id });
  }
}

// ✅ Good - Explicitly void it if fire-and-forget
@Injectable()
class UserService {
  async deleteUser(id: string): Promise<void> {
    void this.logAudit(id); // Intentional fire-and-forget
  }
}
```

#### Require Await in Async Functions
```typescript
// ❌ Bad
async function getUser(): Promise<User> {
  return { id: '1', name: 'John' }; // No await needed
}

// ✅ Good - Remove async if not needed
function getUser(): Promise<User> {
  return Promise.resolve({ id: '1', name: 'John' });
}

// ✅ Good - Use async when you await
async function getUser(): Promise<User> {
  return await this.userRepository.findOne('1');
}
```

### 8. Null/Undefined Safety

#### Strict Boolean Expressions
```typescript
// ❌ Bad
function validateUser(user: User | null) {
  if (user) { // Implicit truthiness check
    return true;
  }
  return false;
}

// ✅ Good - Explicit null check
function validateUser(user: User | null): boolean {
  if (user !== null) {
    return true;
  }
  return false;
}

// ✅ Good - With optional chaining
function getUserName(user: User | null): string | null {
  return user?.name ?? null;
}
```

#### Prefer Nullish Coalescing
```typescript
// ❌ Bad
const userName = user.name || 'Anonymous';

// ✅ Good - Use ?? for null/undefined only
const userName = user.name ?? 'Anonymous';

// Difference:
const name1 = '' || 'default'; // 'default' (empty string is falsy)
const name2 = '' ?? 'default'; // '' (empty string is not nullish)
```

#### No Non-Null Assertions
```typescript
// ❌ Bad
function getUser(id: string): User {
  return users.find(u => u.id === id)!; // Dangerous!
}

// ✅ Good - Handle null case
function getUser(id: string): User | null {
  return users.find(u => u.id === id) ?? null;
}

// ✅ Good - Throw if required
function getUser(id: string): User {
  const user = users.find(u => u.id === id);
  if (user === undefined) {
    throw new NotFoundException('User not found');
  }
  return user;
}
```

### 9. Type Imports

#### Consistent Type Imports
```typescript
// ❌ Bad
import { User, CreateUserDto } from './user';

// ✅ Good - Separate type imports
import type { User } from './user';
import { CreateUserDto } from './user';

// ✅ Good - Or inline type imports
import { type User, CreateUserDto } from './user';
```

### 10. Code Quality

#### No Magic Numbers
```typescript
// ❌ Bad (if enabled)
setTimeout(() => {}, 3600000);

// ✅ Good - Named constants
const ONE_HOUR_MS = 60 * 60 * 1000;
setTimeout(() => {}, ONE_HOUR_MS);

// ✅ Good - With enum
enum Timeout {
  ONE_HOUR = 3600000
}
setTimeout(() => {}, Timeout.ONE_HOUR);
```

#### Prefer Template Literals
```typescript
// ❌ Bad
const message = 'User ' + user.name + ' logged in';

// ✅ Good
const message = `User ${user.name} logged in`;
```

#### Strict Equality
```typescript
// ❌ Bad
if (user.id == '123') {
  // ...
}

// ✅ Good
if (user.id === '123') {
  // ...
}
```

## File-Specific Overrides

### Test Files (`*.spec.ts`, `*.test.ts`)
- Relaxed type safety (any allowed as warning)
- Console statements allowed
- Empty functions allowed
- No explicit return types required

### DTOs (`*.dto.ts`)
- No explicit function return types (class-validator decorators)
- No public accessibility modifier required

### Entities (`*.entity.ts`)
- No public accessibility modifier required (TypeORM decorators)

### Decorators (`*.decorator.ts`)
- Any type allowed as warning (metadata handling)
- No explicit module boundary types

### Migrations (`*.migration.ts`)
- Any type allowed as warning
- Console statements allowed
- No naming convention enforcement

### Config Files
- Any type allowed as warning
- Console statements allowed
- Require imports allowed

### Scripts & Seeds
- Console statements allowed
- Require imports allowed

## Common Patterns

### NestJS Service Example
```typescript
import { Injectable, Logger, NotFoundException } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import type { Repository } from 'typeorm';
import type { User } from './user.entity';
import type { CreateUserDto } from './dto/create-user.dto';

@Injectable()
export class UserService {
  private readonly logger = new Logger(UserService.name);

  constructor(
    @InjectRepository(User)
    private readonly userRepository: Repository<User>,
  ) {}

  public async findAll(): Promise<User[]> {
    this.logger.log('Fetching all users');
    return this.userRepository.find();
  }

  public async findOne(id: string): Promise<User> {
    const user = await this.userRepository.findOne({ where: { id } });
    if (user === null) {
      throw new NotFoundException(`User with ID ${id} not found`);
    }
    return user;
  }

  public async create(createUserDto: CreateUserDto): Promise<User> {
    const user = this.userRepository.create(createUserDto);
    return this.userRepository.save(user);
  }

  private validateEmail(email: string): boolean {
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
  }
}
```

### NestJS Controller Example
```typescript
import { Controller, Get, Post, Body, Param, Logger } from '@nestjs/common';
import type { User } from './user.entity';
import { UserService } from './user.service';
import { CreateUserDto } from './dto/create-user.dto';

@Controller('users')
export class UserController {
  private readonly logger = new Logger(UserController.name);

  constructor(private readonly userService: UserService) {}

  @Get()
  public async findAll(): Promise<User[]> {
    return this.userService.findAll();
  }

  @Get(':id')
  public async findOne(@Param('id') id: string): Promise<User> {
    return this.userService.findOne(id);
  }

  @Post()
  public async create(@Body() createUserDto: CreateUserDto): Promise<User> {
    this.logger.log(`Creating user: ${createUserDto.email}`);
    return this.userService.create(createUserDto);
  }
}
```

## Migration Tips

### 1. Fix Imports First
```bash

npx eslint src --config .eslintrc.extended.json --fix
```

### 2. Add Return Types
Use TypeScript's inference to help:
```typescript
// Hover over function in VSCode to see inferred type
function getUser(id: string) { // Shows: Promise<User | null>
  return this.userRepository.findOne(id);
}

// Add the inferred type
function getUser(id: string): Promise<User | null> {
  return this.userRepository.findOne(id);
}
```

```typescript
// Find all any types
// Search for: ": any"
```

## Performance Considerations

### Large Codebases
- ESLint with type-checking can be slow
- Use `--cache` flag to speed up subsequent runs
- Consider using `--max-warnings 0` in CI

```bash
# Fast development linting
npx eslint src --cache --config .eslintrc.extended.json

# Strict CI linting
npx eslint src --max-warnings 0 --config .eslintrc.extended.json
```

### IDE Integration

#### VSCode Settings
```json
{

    "overrideConfigFile": ".eslintrc.extended.json"

  "eslint.validate": [
```

## Disabling Rules

### Disable for specific lines
```typescript
// eslint-disable-next-line @typescript-eslint/no-explicit-any
function legacy(data: any): any {
  return data;
}
```

### Disable for entire file
```typescript
/* eslint-disable @typescript-eslint/no-explicit-any */
// Legacy migration file
```

### Disable for specific sections
```typescript
/* eslint-disable @typescript-eslint/no-explicit-any */
function legacy1(data: any): any { return data; }
function legacy2(data: any): any { return data; }
/* eslint-enable @typescript-eslint/no-explicit-any */
```

## Benefits

1. **Type Safety**: Catch type errors at compile time
2. **Code Quality**: Consistent patterns across codebase
3. **Maintainability**: Easier to understand and modify
4. **Performance**: Fewer runtime errors
5. **Developer Experience**: Better IDE autocomplete and hints
6. **NestJS Best Practices**: Enforces framework conventions

## Common Issues

### Issue: Too many errors
**Solution**: Migrate gradually, one module at a time

### Issue: ESLint is slow
**Solution**: Use `--cache` flag and consider `project-service` mode

### Issue: Rule conflicts with Prettier
**Solution**: This config includes prettier integration, no conflicts

### Issue: Can't use any type for valid reasons
**Solution**: Use `unknown` or create proper types, or disable for specific line
